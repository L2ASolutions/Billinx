# Testing

Moved out of `CLAUDE.md` on 2026-07-21 (content unchanged, verbatim) to keep the
main file within a manageable size. Covers per-module backend test coverage,
frontend unit-test infrastructure, and the Playwright E2E suite design.

## Running the suites

```bash
# Backend unit tests (Jest)
npm test                                    # from repo root
npm run test:cov                            # with coverage

# Frontend unit tests (Vitest)
cd apps/web && npm test
cd apps/web && npm run test:coverage

# E2E tests (Playwright) — needs a locally running backend (port 3000) and
# web app (port 3001, `npm run dev -- -p 3001`), with Postgres + Redis up
cd apps/web && npm run test:e2e
cd apps/web && npm run test:e2e:ui          # interactive UI mode
cd apps/web && npm run test:e2e:headed      # headed browser
cd apps/web && npm run test:e2e:report      # view last HTML report

# RLS isolation integration test
npm run test:integration
```

  - **Auth architecture — not a per-test UI login.** `POST /v1/auth/login` (and `/v1/auth/mfa/challenge`) sit behind `AuthRateLimitGuard`, a strict 5-requests/15-minutes-**per-IP** bucket shared across login/register/forgot-password. Every Playwright worker on one CI runner shares an IP, so 5 independent journeys each doing their own full UI login (2 calls for the OWNER journeys — password step + MFA challenge) would exceed that budget on a single run, before even counting Playwright's `retries: 1`. Confirmed live: hit "Too many authentication attempts" repeatedly during local iteration. Fixed with Playwright's own recommended pattern — a `setup` project (`e2e/auth.setup.ts`) that logs in exactly once per role via the real login form (`e2e/helpers/auth.ts`'s `loginAsOwner()`/`loginAsViewer()` — this is still genuine UI+MFA-flow coverage, just run once instead of once per test) and saves the resulting localStorage-token state to `e2e/.auth/{owner,viewer}.json` (gitignored). Every journey file then does `test.use({ storageState: 'e2e/.auth/owner.json' })` and starts already authenticated — 2 real login calls total for the whole suite, regardless of how many journeys or retries run.
  - **OWNER logins always require MFA** (`UserService.login()`'s `isPrivileged` check — OWNER/ADMIN roles get `mfaSetupRequired`/`mfaRequired` unconditionally), so a naive `loginAsOwner()` would land on `/mfa/setup` (QR-code enrolment), not `/dashboard`. `scripts/seed-dev-users.ts` now pre-enables MFA on the seeded owner test user with a **fixed, hardcoded, E2E-only TOTP secret** (`E2ETESTOWNERSECRETXYZABCDEFGH23` — must stay byte-identical between that script and `e2e/helpers/auth.ts`/`e2e/helpers/totp.ts`), encrypted the same way `MfaService.setupMfa()` does (`CredentialService.encrypt()` with the tenant/user id as the key-derivation input). `e2e/helpers/totp.ts` computes the current 6-digit code from that same secret — ported byte-for-byte from `MfaService`'s own `totpGenerate()`/`base32Decode()` (RFC 6238, SHA1/6-digits/30s) rather than pulling in a TOTP library, so it can never silently drift from what the server actually verifies against. `loginAsOwner()` fills the real 6-digit challenge UI with the computed code — genuine MFA-flow coverage, not a bypass.

  - **A real `Page.waitForURL()` footgun caught via live verification, not by reading the code.** `waitForURL` resolves immediately if the *current* URL already matches the pattern — it does not wait for a fresh navigation. `invoice-round-trip.spec.ts` was on `/invoices/new` when it called `waitForURL(/\/invoices\/[^/]+$/)` expecting to wait for the post-submit redirect to the new invoice's detail page — but `"new"` itself satisfies `[^/]+`, so the wait resolved instantly against the *current* URL, and the "invoice id" extracted from it was literally the string `"new"`, which then 404'd on the public pay page (`GET /api/v1/invoices/pay/new`) several assertions later. Confirmed via a Playwright trace showing that exact request. Fixed by anchoring both the wait and the extraction to a UUID shape (`/\/invoices\/[0-9a-f-]{36}$/i`) instead of a bare non-slash-chars pattern; `role-access.spec.ts` had the same loose pattern (not actually exploitable there, since its preceding page never matches it, but tightened for consistency).

  - **Related, parallel RBAC gap found but *not* fixed here (flagged, out of scope):** the invoice detail page's `canCancel`/`canRecordPayment`/`canSendReminder` (`invoices/[id]/page.tsx`) are computed purely from invoice **status**, not role — a VIEWER can currently see "Cancel invoice"/"Record payment"/"Send reminder" buttons on an eligible invoice and would only be rejected server-side on click, the same class of gap as the Create-invoice one above. `role-access.spec.ts`'s literal "no Submit button on any invoice detail page" assertion passes regardless (there's no such button on that page for *any* role — submission only happens from the New Invoice form's preview step), so it doesn't currently catch this. Worth a follow-up PR extending the same `canCreateInvoice`-style pattern to these three buttons.
